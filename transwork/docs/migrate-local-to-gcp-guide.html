0 of 0 done

Migrate Gressio-server + Logto from your laptop to Google Cloud

Lift-and-shift your two local Docker Compose stacks (Logto + Gressio-server) onto the existing GCP VM 34.142.134.146, behind HTTPS on your gressio.ai domain. The desktop client stays on your computer — you only re-point it at the cloud. Fresh start (no data carried over from the old VM image or your laptop). Estimated total time: 60–120 minutes.

What you're building

Nginx — TLS edge (host + Certbot) Gressio-server — redeploy current version Logto — new on this VM
Your starting point (assumptions) This guide assumes: (1) the VM 34.142.134.146 already exists, is yours, and currently runs an old Gressio-server via Docker Compose; (2) api.gressio.ai already resolves to that VM; (3) you own the gressio.ai DNS zone and can add records; (4) you're doing a fresh start — the old server's users/tokens/billing and any local Logto data can be discarded. If any of these is wrong, stop and adjust before running the destructive steps in Step 4.
Why HTTPS, not raw IP Because you have a real domain, this guide terminates TLS at your host Nginx (certs from Certbot) and serves everything over HTTPS. Logto's OIDC issuer must be a single stable URL, and browsers + the desktop OAuth flow behave far better over HTTPS. The container ports (3000/3001/3002) are published only on 127.0.0.1 — Nginx proxies to them, and only its :80/:443 face the internet.

2
Open the firewall for HTTP/HTTPS (80 + 443)
~5 min
›

With Nginx as the only public entry point, you just need ports :80 (for Let's Encrypt's challenge + redirect) and :443 (HTTPS) open. The app ports 3000/3001/3002 are reached only via 127.0.0.1 on the host.

gcloud compute firewall-rules create allow-https-gressio \
  --direction=INGRESS \
  --action=ALLOW \
  --rules=tcp:80,tcp:443 \
  --source-ranges=0.0.0.0/0 \
  --description="Nginx HTTP+HTTPS for gressio.ai"
Close the old :3000 rule once Nginx HTTPS is live The old setup likely has port 3000 open to the world. Leave it for now so you don't lock yourself out mid-migration, but in Step 11 you'll remove that rule so the only way in is HTTPS through Nginx.

Make sure the VM also has a static external IP so 34.142.134.146 survives a reboot:

gcloud compute addresses list   # the VM's IP should show TYPE=EXTERNAL, not ephemeral

3
Confirm Nginx and install Certbot
~5 min
›

Your VM already runs host-installed Nginx (that's how api.gressio.ai is served today). Confirm it's healthy, then install Certbot with the Nginx plugin so it can issue and auto-renew Let's Encrypt certificates.

sudo systemctl status nginx --no-pager   # should be "active (running)"
nginx -v
sudo nginx -t                            # current config is valid

Install Certbot (Debian/Ubuntu). If it's already present this is a no-op:

sudo apt update
sudo apt install -y certbot python3-certbot-nginx
No shared Docker network needed Because Nginx runs on the host (not in a container), it reaches each service over a published localhost port — proxy_pass http://127.0.0.1:<port>. There's nothing to wire on the Docker side; the containers just publish their ports to 127.0.0.1 (Steps 5–6) and Nginx does the rest.

5
Stand up Logto (its own stack)
~10 min
›

Logto gets its own directory and its own dedicated Postgres, completely separate from Gressio-server. Note the ENDPOINT/ADMIN_ENDPOINT are the public HTTPS names — Logto bakes these into its OIDC discovery document, so they must be the real outside URLs, not localhost.

sudo mkdir -p /opt/logto && sudo chown $USER:$USER /opt/logto
cd /opt/logto
cat > docker-compose.yml <<'EOF'
services:
  logto:
    image: svhd/logto:latest
    container_name: logto
    restart: unless-stopped
    depends_on:
      logto-postgres:
        condition: service_healthy
    ports:
      - "127.0.0.1:3001:3001"   # Core / OIDC — host Nginx proxies here
      - "127.0.0.1:3002:3002"   # Admin Console — host Nginx proxies here
    environment:
      TRUST_PROXY_HEADER: 1
      DB_URL: postgres://logto:CHANGE_ME_DB_PASS@logto-postgres:5432/logto
      ENDPOINT: https://auth.gressio.ai
      ADMIN_ENDPOINT: https://admin.gressio.ai
    entrypoint: ["sh", "-c", "npm run cli db seed -- --swe && npm start"]

  logto-postgres:
    image: postgres:16-alpine
    container_name: logto-postgres
    restart: unless-stopped
    environment:
      POSTGRES_USER: logto
      POSTGRES_PASSWORD: CHANGE_ME_DB_PASS
      POSTGRES_DB: logto
    volumes:
      - logto_pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U logto -d logto"]
      interval: 5s
      timeout: 5s
      retries: 10

volumes:
  logto_pgdata:
EOF
Replace the password in both places Generate a strong one and substitute it for CHANGE_ME_DB_PASS: 
PASS=$(openssl rand -hex 24)
sed -i "s/CHANGE_ME_DB_PASS/$PASS/g" docker-compose.yml
docker compose up -d
docker compose logs -f logto    # Ctrl-C once you see "Core app is running"
Ports bound to 127.0.0.1 only Publishing as 127.0.0.1:3001 (not 0.0.0.0:3001) means only the host — i.e. your Nginx — can reach Logto. The ports never face the public internet directly; the only way in is the HTTPS name Nginx serves.

Unlike a containerized reverse proxy, host Nginx needs no Docker-network wiring. The base docker-compose.yml already publishes 3000:3000 on the host, and Nginx will proxy to 127.0.0.1:3000. Optional hardening If you'd rather not have new-api bound to 0.0.0.0:3000 at all (relying only on the firewall), add a one-line override that re-maps the port to 127.0.0.1:3000:3000 and include it with an extra -f. Not required — the firewall change in Step 11 already closes external access.

Fill in the Transwork env file

The overlay needs a Google Cloud Storage service-account key (for batch ASR uploads) and the server-side ElevenLabs key. Copy the example and edit it:

cp transwork/transwork.env.example transwork/transwork.env

7
Configure Nginx server blocks + issue certs with Certbot
~15 min
›

Add (or update) one Nginx server block per hostname, each proxying to the matching localhost port, then let Certbot upgrade them to HTTPS. Start with plain HTTP so Certbot's HTTP-01 challenge can succeed on port 80.

api.gressio.ai may already have a block Since Nginx already serves api.gressio.ai, you may only need to add the auth and admin blocks and confirm api proxies to 127.0.0.1:3000. Check /etc/nginx/sites-enabled/ first — don't define the same server_name twice.

Create the site config (HTTP only for now — Certbot adds the TLS lines next):

sudo tee /etc/nginx/sites-available/gressio.conf > /dev/null <<'EOF'
server {
    listen 80;
    server_name api.gressio.ai;
    client_max_body_size 200M;   # desktop app uploads audio for batch ASR

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_buffering off;          # SSE: stream chat tokens without buffering
        proxy_read_timeout 3600s;
    }
}

server {
    listen 80;
    server_name auth.gressio.ai;

    location / {
        proxy_pass http://127.0.0.1:3001;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}

server {
    listen 80;
    server_name admin.gressio.ai;

    location / {
        proxy_pass http://127.0.0.1:3002;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
}
EOF
Two settings that bite if you skip them client_max_body_size on api — Nginx's 1 MB default rejects batch-ASR audio uploads with 413 Request Entity Too Large. And X-Forwarded-Proto $scheme on every block — Logto (with TRUST_PROXY_HEADER=1) and new-api rely on it to know the request arrived over HTTPS, otherwise they generate http:// redirect URLs and the OAuth flow breaks.

Enable the site and reload:

sudo ln -sf /etc/nginx/sites-available/gressio.conf /etc/nginx/sites-enabled/gressio.conf
sudo nginx -t          # must pass
sudo systemctl reload nginx

Now issue certs for all three names. The nginx plugin edits the config in place to add listen 443 ssl and an HTTP→HTTPS redirect:

sudo certbot --nginx \
  -d api.gressio.ai \
  -d auth.gressio.ai \
  -d admin.gressio.ai \
  --redirect --agree-tos -m you@gressio.ai --no-eff-email
Certs need DNS resolving + port 80 reachable Certbot's HTTP-01 challenge hits http://<name>/.well-known/acme-challenge/.... If the auth/admin A records from Step 1 haven't propagated, issuance fails — dig auth.gressio.ai +short should return 34.142.134.146 first. Renewal is automatic via the installed certbot.timer; confirm with sudo certbot renew --dry-run.

Smoke-test from your laptop:

curl -I https://api.gressio.ai
curl https://auth.gressio.ai/oidc/.well-known/openid-configuration | head
# issuer should read "https://auth.gressio.ai"

10
Point the desktop client at the cloud (baked-in URL)
~10 min
›

How the client finds the server changed with the recent dev-mode / locked-down-config work. The production server URL is now compiled into the binary — it's no longer read from a config file on the end user's machine.

A release build needs no per-machine config The constant PRODUCTION_SERVER_BASE_URL in src-tauri/src/config/mod.rs is baked in and already set to https://api.gressio.ai. Release builds ship without api_keys.toml, and the embedded default_config.json already routes through the server relay (routingMode = transwork_server, provider = elevenlabs, useTransworkServer = true). So once a user installs the build and signs in, the client talks to your cloud automatically — nothing to edit on their machine.

A. Release build (production / end users)

The only "re-pointing" knob is the compiled constant. Confirm it matches this deployment:

// transwork/src-tauri/src/config/mod.rs
pub const PRODUCTION_SERVER_BASE_URL: &str = "https://api.gressio.ai";

It already reads https://api.gressio.ai, so just build and distribute the installer:

cd transwork
npm run tauri build
Changing the domain later means a rebuild, not a config edit Because the URL is compiled in and release builds don't read api_keys.toml (its load path resolves to the build machine's source tree, which doesn't exist on a user's install), you can't re-point a shipped client by dropping a config file next to it. To move domains, edit the constant and ship a new build. Also update PRODUCTION_OPENCODE_MODEL in the same file if the agent model changes.

B. Dev build pointing at the cloud (testing from source)

If you're running npm run tauri dev against the cloud, set dev_mode = true at the top of the git-ignored transwork/api_keys.toml. That unlocks the full Settings UI and makes the canonical config.json editable. The server URL already defaults to the baked production value, so overrides are optional:

# transwork/api_keys.toml
dev_mode = true

[asr.elevenlabs]
# Optional — defaults to the baked PRODUCTION_SERVER_BASE_URL already.
# server_base_url = "https://api.gressio.ai"
# user_token = ""        # leave empty; the login token comes from the keyring

[opencode]
use_transwork_server = true
# server_base_url = "https://api.gressio.ai"   # optional override
Token comes from sign-in, not the file Leave user_token empty in either mode. After "Sign in with browser", the 90-day desktop token lives in Windows Credential Manager and is used automatically — api_keys.toml never carries it.

Lock down the old port

Now that HTTPS works, remove the old public :3000 rule so the only entry is Nginx:

gcloud compute firewall-rules list                       # find the old rule's name
gcloud compute firewall-rules delete <old-3000-rule>     # confirm before deleting
Optional: restrict the admin console admin.gressio.ai is internet-reachable (password-protected, but still). To limit it to your IP, add allow YOUR.IP; then deny all; inside the location / of the admin.gressio.ai server block, then sudo nginx -t && sudo systemctl reload nginx.

Daily backups

Two databases now: Gressio-server's Postgres (users/tokens/billing) and Logto's Postgres (identities). Snapshot both nightly:

cat > /opt/backup.sh <<'EOF'
#!/bin/bash
set -e
DEST=/opt/backups; mkdir -p $DEST
DATE=$(date +%F-%H%M)
docker exec postgres pg_dumpall -U root | gzip > $DEST/gressio-$DATE.sql.gz
docker exec logto-postgres pg_dumpall -U logto | gzip > $DEST/logto-$DATE.sql.gz
find $DEST -name "*.sql.gz" -mtime +14 -delete
EOF
chmod +x /opt/backup.sh
(crontab -l 2>/dev/null; echo "0 3 * * * /opt/backup.sh") | crontab -
Adjust the container/user names postgres / root are the defaults from the server's compose; logto-postgres / logto match Step 5. Run docker ps to confirm the actual names before trusting the cron.
