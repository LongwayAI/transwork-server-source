1
Pre-flight: confirm the VM and Gressio-server
~5 min
›

Before touching anything, SSH in and verify the existing setup is healthy. If the old Gressio-server is broken, we want to know now, not after we've added Logto on top.

gcloud compute ssh <your-vm-name> --zone=<your-zone>
# or: ssh your-user@34.21.152.4

Once inside:

cd ~/transwork-server   # adjust path if different
docker compose ps        # all services should be "running" / "healthy"
docker ps --format "table {{.Names}}\t{{.Status}}\t{{.Ports}}"
df -h /                  # confirm at least ~5 GB free
free -h                  # confirm at least ~2 GB free RAM
Why this matters Logto + its Postgres need ~1 GB RAM and ~500 MB disk. With 8 GB and Gressio-server already running, you've got plenty of room — but a quick check now avoids the "out of memory" surprise mid-deploy.

Checklist

2
Open GCP firewall ports 3001 and 3002
~5 min
›

Logto runs on two ports: :3001 for the OIDC endpoints (the browser hits this during sign-in) and :3002 for the admin console. Both need to be reachable from your laptop.

Open the GCP Console → VPC network → Firewall, or use gcloud:

gcloud compute firewall-rules create allow-logto-alpha \
  --direction=INGRESS \
  --action=ALLOW \
  --rules=tcp:3001,tcp:3002 \
  --source-ranges=0.0.0.0/0 \
  --description="Logto core (3001) + admin (3002) for alpha"
Tighten this later 0.0.0.0/0 means "anyone on the internet." That's fine for alpha — Logto's admin is password-protected and the OIDC endpoints need to be public anyway. But before public launch, restrict :3002 (admin) to your home/office IP only.

Find your home IP at whatismyipaddress.com if you want to tighten it now:

# Optional: admin port restricted to your IP only
gcloud compute firewall-rules create allow-logto-admin \
  --direction=INGRESS --action=ALLOW \
  --rules=tcp:3002 \
  --source-ranges=YOUR.HOME.IP.HERE/32

3
Stand up Logto via Docker Compose
~10 min
›

Logto gets its own directory and its own dedicated Postgres — completely separate from Gressio-server's Postgres. This keeps the two schemas independent and means an upgrade to one can't break the other.

On the VM:

sudo mkdir -p /opt/logto
sudo chown $USER:$USER /opt/logto
cd /opt/logto

Create docker-compose.yml:

cat > docker-compose.yml <<'EOF'
services:
  logto:
    image: svhd/logto:latest
    container_name: logto
    restart: unless-stopped
    depends_on:
      logto-postgres:
        condition: service_healthy
    ports:
      - "3001:3001"   # Core / OIDC endpoints
      - "3002:3002"   # Admin Console
    environment:
      TRUST_PROXY_HEADER: 1
      DB_URL: postgres://logto:CHANGE_ME_DB_PASS@logto-postgres:5432/logto
      ENDPOINT: http://34.21.152.4:3001
      ADMIN_ENDPOINT: http://34.21.152.4:3002
    entrypoint: ["sh", "-c", "npm run cli db seed -- --swe && npm start"]

  logto-postgres:
    image: postgres:16-alpine
    container_name: logto-postgres
    restart: unless-stopped
    environment:
      POSTGRES_USER: logto
      POSTGRES_PASSWORD: CHANGE_ME_DB_PASS
      POSTGRES_DB: logto
    volumes:
      - logto_pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U logto -d logto"]
      interval: 5s
      timeout: 5s
      retries: 10

volumes:
  logto_pgdata:
EOF
Replace the password Search/replace CHANGE_ME_DB_PASS with a strong random string (openssl rand -hex 24) in both places. The Logto Postgres is internal-only (not exposed to the host), but rotating the default still matters in case you later expose it for debugging.

Generate a password and bring it up:

PASS=$(openssl rand -hex 24)
sed -i "s/CHANGE_ME_DB_PASS/$PASS/g" docker-compose.yml

docker compose up -d
docker compose logs -f logto    # Ctrl-C once you see "Core app is running at http://localhost:3001"
Why the explicit ENDPOINT env vars? Logto generates OIDC discovery URLs (issuer, jwks_uri, etc.) using ENDPOINT. If you don't set it, Logto advertises http://localhost:3001 in its .well-known/openid-configuration, and the browser-side redirect to localhost fails on your laptop. Setting it to the VM's public IP makes the discovery doc consistent.

Smoke-test from your laptop:

curl http://34.21.152.4:3001/oidc/.well-known/openid-configuration | head
# Should return JSON with "issuer": "http://34.21.152.4:3001"

First-time Logto admin setup
~5 min
›

Open the admin console in your browser:

http://34.21.152.4:3002

Logto walks you through a first-run wizard:

  1. Click Create account.
  2. Pick a username (this is the Logto admin username, not your Gressio users). Save it in your password manager.
  3. Pick a strong password. Save it.
  4. Logto may ask a few questions ("how will you use Logto?", "what languages?"). Answer or skip — doesn't affect functionality.
This admin account is the master key Whoever has it can change every sign-in setting and see all users. Treat it like a root password — long, unique, in a vault, no sharing.

Pull the latest Gressio-server code and redeploy
~5–10 min
›

The VM has an older version. Recent commits added desktop OIDC support and the max=20 username/displayName fix. Pull and rebuild:

cd ~/transwork-server
git fetch origin
git log HEAD..origin/main --oneline    # preview what's coming in
git pull origin main

Rebuild the image and restart, using the Gressio overlay (matches what's documented in transwork/transwork.env.example):

docker compose \
  --env-file transwork/transwork.env \
  -f docker-compose.yml \
  -f transwork/docker-compose.transwork.yml \
  up -d --build
Existing data is preserved The Postgres volume (pg_data) is unaffected by a rebuild. Your existing users, channels, tokens, billing config all stay. If you're nervous, snapshot the volume first: docker run --rm -v transwork-server_pg_data:/data -v $(pwd):/backup alpine tar czf /backup/pg-backup-$(date +%F).tar.gz /data

Watch logs for a clean startup:

docker compose logs -f new-api    # Ctrl-C once you see "Server started at :3000"

9
Update the desktop client and run the end-to-end test
~10 min
›

The desktop client (transwork/ repo) needs to know about the production server URL. Depending on how the client reads it:

  • If hard-coded in src/lib/api.ts: change the base URL to http://34.21.152.4:3000, rebuild the installer (npm run tauri build).
  • If user-configurable in Settings: just change it in the running app.
  • If read from an env file at build time: update that file and rebuild.

Test flow

  1. Open the (updated) desktop client.
  2. Click Sign in with OIDC (or whatever your sign-in button says).
  3. System browser opens to http://34.21.152.4:3001/oidc/auth?... → Logto sign-in screen.
  4. Click Create account. Enter email + password. Submit.
  5. Browser redirects to http://127.0.0.1:3000/api/desktop-login/callback?code=... → desktop client picks it up.
  6. Desktop client window comes to the front, signed in.
  7. Click the profile pop-up — it should show the email you just used.
Bonus verification Back in Logto admin → User management, you should see the new user row with the email you just used and no username field populated. In Gressio-server admin → Users, the same user should appear with an auto-generated username like oidc_a1b2c3d4... and the email filled in. That's expected — the desktop login handler auto-generates the username, since you removed the prompt in step 6.

SymptomCheck
"Connection refused" when browser hits :3001Firewall rule from step 2 didn't apply; gcloud compute firewall-rules list
"Invalid redirect URI"Step 5 — the exact URI in the request must be one of the URIs you registered, character-for-character
Logto sign-in works but loops back / no user created in GressioStep 7 — Client Secret has invisible whitespace, or endpoint URLs typo'd. Re-check.
Logs show "discovery doc points to localhost"Step 3 — ENDPOINT env var wasn't picked up. docker compose down && docker compose up -d to reload it.

Backup, logs, and "what's next"
~5 min
›

Daily backups (set this up before inviting any alpha user)

Two Postgres volumes to back up: transwork-server_pg_data (users, channels, billing) and logto_logto_pgdata (auth identities). A simple cron job:

cat > /opt/backup.sh <<'EOF'
#!/bin/bash
set -e
DEST=/opt/backups
mkdir -p $DEST
DATE=$(date +%F-%H%M)
docker exec postgres pg_dumpall -U root | gzip > $DEST/transwork-$DATE.sql.gz
docker exec logto-postgres pg_dumpall -U logto | gzip > $DEST/logto-$DATE.sql.gz
# Keep last 14 days
find $DEST -name "*.sql.gz" -mtime +14 -delete
EOF
chmod +x /opt/backup.sh

# Run daily at 03:00
(crontab -l 2>/dev/null; echo "0 3 * * * /opt/backup.sh") | crontab -

Watching logs

# Gressio-server
cd ~/transwork-server && docker compose logs -f --tail=100 new-api

# Logto
cd /opt/logto && docker compose logs -f --tail=100 logto

What to do after alpha goes well

When you're ready to harden this for real users, these are the next steps (each is its own mini-project):

  1. Get a domain ($1–3/year on Cloudflare). Point an A record at 34.21.152.4.
  2. Add Caddy as a reverse proxy in front of both services. Auto-issues Let's Encrypt certs. Now both endpoints are HTTPS.
  3. Enable Google login in Logto: add the Google connector, register the OAuth client in Google Cloud Console using your new domain.
  4. Restrict Logto admin port (3002) to your IP only.
  5. Move to Cloud SQL once you have more than ~20 users — managed backups, no VM disk pressure.
  6. Static external IP if you haven't already, so the VM IP doesn't change after a reboot.
Reach out for the hardening pass Each of those steps is small — domain + Caddy is ~30 minutes, Google login is ~15 minutes once Caddy is in place. Tackle them after alpha confirms the product idea is worth investing in.
