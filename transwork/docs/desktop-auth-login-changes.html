What this branch ships

Two feature areas, both reachable through the existing API gateway and both wired in transwork/routes.go:

1. Desktop login bridge /api/desktop-login

A three-leg flow that lets a Tauri desktop app authenticate a user through Logto (OIDC) without ever holding the OAuth client secret. The server browser receives the OIDC code, then bounces a single-use bootstrap code to http://127.0.0.1:<port> on the user's machine, which the desktop app exchanges for a 90-day bearer token row in the existing Token model.

desktop_login.go · desktop_login_store.go (443 lines + 244 test lines)

2. Auth endpoints for the desktop app /api/auth

Three small handlers behind middleware.TokenAuth() so the desktop app can keep its session UI in sync: GET /check returns identity + quota state, POST /logout deletes the token row, POST /redeem credits an invite/redemption code's quota to the current user.

auth_check.go · auth_logout.go · auth_redeem.go (131 lines + 136 test lines)

Why an overlay instead of patching upstream

Every new file lives under transwork/handler/; the only "wiring" line into the upstream router is the existing RegisterAPIRoutes call in transwork/routes.go — itself an overlay file. That's intentional: when QuantumNous/new-api upstream releases a new version, this branch will rebase with zero conflicts because no upstream files were touched.

One temptation was avoided in commit 2d91814a: the upstream oauth.OIDCProvider.ExchangeToken hardcodes redirect_uri to /oauth/oidc, which would have failed RFC 6749 §4.1.3's exact-match rule. Rather than monkey-patch the upstream method, an inline exchangeOidcCodeForDesktop helper was added next to the desktop handler.

End-to-end flow (click "Play" to step through)

Six actors — you can see at a glance which calls cross which trust boundary.

1

Desktop app picks a loopback port

Desktop client → POST /api/desktop-login/start

Body: { loopbackPort: 53241 } — must be ≥1024. Server returns authUrl and a random 16-byte hex state; both are stored in an in-memory TTL map (5 min).

2

System browser opens authUrl

Desktop client → OS browser

The Tauri app shells out the URL to the OS, then starts a tiny loopback HTTP server on the chosen port. User completes Logto sign-in (or cancels).

3

Logto 302s back to the API gateway

Logto → GET /api/desktop-login/callback?code&state

The state is consumed (single-use). If anything has gone wrong server-side (OIDC disabled, code missing, exchange failed, user disabled) the error is forwarded to the loopback URL — the desktop app handles all error UI.

4

Server exchanges code for user info

Server → Logto token endpoint

exchangeOidcCodeForDesktop() POSTs to tokenEndpoint with redirect_uri = …/api/desktop-login/callback (matching what was sent during /authorize). Then provider.GetUserInfo resolves identity. findOrCreateOidcUserForDesktop finds or creates the local User.

5

Server 302s to the loopback with a bootstrap code

Server → http://127.0.0.1:<port>/desktop-login/callback

32-byte hex bootstrap code (256 bits of entropy) stored in another in-memory TTL map. The browser tab now closes — control is back inside the user's machine.

6

Desktop app exchanges bootstrap → 90-day token

Desktop client → POST /api/desktop-login/exchange

Bootstrap code is consumed (single-use). Server inserts a new Token row: UnlimitedQuota: true, Group: "", key = 24 random bytes (48 hex chars), expiry = now + 90 d. Returns {token, expiresAt, userId}. Desktop app stores the token in OS keychain.

Desktop-login endpoints (no auth — they bootstrap auth)

All three are public; safety comes from short-TTL random codes and OIDC.

POST /api/desktop-login/start public ▶

Request

{
  "loopbackPort": 53241   // 1024–65535
}

200 Response

{
  "authUrl": "https://logto.../oidc/auth?...",
  "state":   "<32 hex chars>"
}

Errors

  • 400 — invalid JSON / port out of range
  • 500 — RNG failure (can't generate state)
  • 503 — OIDC disabled or unconfigured

Notes

State is stored in desktopLoginStates with the loopback URL attached; consumed in step 3.

GET /api/desktop-login/callback?code&state public ▶

Behavior

Validates state, exchanges the OIDC code, finds-or-creates the user, mints a bootstrap code, and HTTP-302s the browser to the user's loopback URL.

Success redirect

302 → http://127.0.0.1:53241/desktop-login/callback
       ?code=<64 hex>&state=<same as start>

Error redirect (always to loopback)

302 → http://127.0.0.1:53241/desktop-login/callback
       ?error=<reason>&state=...

Reasons: missing_code, oidc_not_enabled, exchange_failed, userinfo_failed, server_error, user_disabled, plus any IdP-side error code passed through verbatim. The desktop app owns the error UI.

POST /api/desktop-login/exchange public ▶

Request

{ "code": "<bootstrap from callback>" }

200 Response

{
  "token":     "<48 hex chars>",
  "expiresAt": 1768000000,
  "userId":    42
}

Errors

  • 400 — missing/invalid/expired bootstrap code
  • 403 — user no longer exists or is disabled
  • 500 — RNG / DB failure

Token row

Inserted into existing Token table. UnlimitedQuota: true (per-token cap is absent — quota lives on User.Quota), expiry now + 90 d, name "Desktop Login".

Auth endpoints (all behind TokenAuth())

GET /api/auth/check bearer token ▶

Response

{
  "userId":         42,
  "email":          "user@example.com",
  "displayName":    "Alex",
  "group":          "default",
  "userQuota":      500000,
  "remainQuota":    null,    // null → no per-token cap
  "unlimitedQuota": true,
  "tokenExpiresAt": 1768000000   // -1 sentinel = unknown / never
}

Why both userQuota & remainQuota

remainQuota is the per-token cap (used for shared/programmatic tokens); for desktop tokens this is null. userQuota is the per-user balance and is the field the desktop app uses to gate the invite-redemption UI when it hits zero.

POST /api/auth/logout bearer token ▶

Deletes the current token row via model.DeleteTokenById(tokenID, userID). Response shape mirrors upstream's {success:true, message:""} so the existing client decoder doesn't need a special case.

POST /api/auth/redeem bearer token ▶

Request

{ "code": "INVITE-ABCD-EFGH" }

200 Response

{ "quotaAdded": 100000 }

Implementation

Pure delegation: model.Redeem(code, userID). The handler is 36 lines including imports — it exists only because the desktop client wants a stable, scoped path under /api/auth rather than coupling to the dashboard's redemption route.

Why the design looks like it does

Each card explains one non-obvious decision and links the commit that captured it.

Bootstrap-code indirectionSECURITY

The browser never sees the long-lived API token. Instead it gets a 32-byte single-use code that the desktop app — running on the same machine — exchanges over a separate HTTPS round-trip. That keeps the long token out of browser history, OS URL logs, and any extension that can read the active tab.

desktop_login.go: bootstrapCodes / ExchangeDesktopLoginBootstrap

Custom OIDC token exchangeRULE 4 / OVERLAY

Upstream oauth.OIDCProvider.ExchangeToken hardcodes redirect_uri to /oauth/oidc. RFC 6749 §4.1.3 requires the redirect_uri at exchange time to match the one at /authorize exactly, so we'd be guaranteed an invalid_grant. We added a parallel helper instead of editing upstream — overlay rule preserved.

commit 2d91814a · exchangeOidcCodeForDesktop()

All errors flow through the loopbackUX

OIDC failures, exchange failures, disabled-user errors — every code path that aborts after state-consumption 302s back to the desktop loopback with ?error=…. The desktop app owns all error UI; the user is never stranded on a server-rendered page in their browser.

redirectErrToLoopback() · commits 5258c8f4, 45c62e1c

In-memory TTL stores, not RedisSCOPE

State and bootstrap codes both have 5-minute TTL and are scoped to a single login attempt — the user is staring at the screen the whole time. A mutex-guarded map with opportunistic GC is plenty; pulling Redis in would have added a deploy dependency for a 5-minute window of data.

desktop_login_store.go (104 lines, 54 test lines)

Random username suffix, not GetMaxUserId()+1CONCURRENCY

Two concurrent OIDC sign-ups would both read the same max-id and produce the same username. randomShortSuffix() uses 8 bytes (64 bits) from crypto/rand — collision probability is vanishing while staying under the 20-char struct-validation limit.

commit 66671cc0 · randomShortSuffix()

Token has no per-token capDATA MODEL

Setting UnlimitedQuota: true on the row means the per-user User.Quota is the only ledger the desktop session draws from. Otherwise the dashboard and desktop would each have a separate balance, which would surprise users and complicate refunds/redemptions.

desktop_login.go: ExchangeDesktopLoginBootstrap

tokenExpiresAt: -1 sentinelAPI CONTRACT

Avoids an omitempty + 0 ambiguity (was the token created at the unix epoch, or do we just not know?). The desktop client treats -1 as "don't show an expiry warning"; any positive value is real.

commit 66671cc0 · auth_check.go

Token key uses no -COMPATIBILITY

Upstream TokenAuth() middleware splits keys on - for legacy reasons. Generating a 24-byte hex key (48 chars, all 0–9a-f) sidesteps the splitter and behaves identically to dashboard-issued tokens.

desktop_login.go: randomHex(24)
